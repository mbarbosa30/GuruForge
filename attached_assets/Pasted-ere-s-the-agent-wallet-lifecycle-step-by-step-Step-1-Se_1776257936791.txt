ere's the agent wallet lifecycle, step by step:

Step 1 — Server boots for the first time

The server checks: does the file agent_wallet.enc exist? If not, it needs to create a wallet from scratch.

Step 2 — Generate a random private key

eth_account.Account.create() runs. This produces two things:

A random private key (a long hex string — this is the "password" to the wallet)
A public address derived from it (like 0x22DaEF60B252... — this is the wallet's identity on the blockchain)
At this point, the private key exists only in memory. Nothing is on disk yet.

Step 3 — Read the server secret

The server pulls SESSION_SECRET from Replit's environment secrets. This is a string you set once — it never changes. Think of it as the master password that protects everything.

Step 4 — Derive an encryption key from the secret

The SESSION_SECRET string is fed through scrypt — a function designed to be slow and memory-intensive on purpose (to make brute-force guessing impractical). It produces a 256-bit key. This derived key is what actually encrypts/decrypts — not the raw secret string.

Step 5 — Encrypt the private key

Using AES-256-GCM (an industry-standard authenticated encryption algorithm):

A random 16-byte IV (initialization vector) is generated — ensures the same private key encrypted twice would produce different results
The private key is encrypted, producing ciphertext + an authentication tag (which detects tampering)
The result is formatted as iv:authTag:ciphertext — three hex strings joined by colons
Step 6 — Save to disk

The encrypted result is written to agent_wallet.enc as a JSON file containing:

The public address (not secret — this is visible on the blockchain anyway)
The encrypted private key (useless without SESSION_SECRET)
Metadata (wallet type, creation info)
The raw private key is discarded from memory. It no longer exists anywhere in plaintext.

Step 7 — Generate recovery shares

The derived encryption key (from Step 4) is split using Shamir Secret Sharing into 2 pieces:

Share A is saved on the server in recovery_data.json
Share B is shown to the user (displayed once in the UI)
Neither share alone can do anything. You need both to reconstruct the encryption key.

Step 8 — Wallet is ready

From this point on, the server boots and just reads the address from agent_wallet.enc. It never decrypts the private key unless it needs to sign a transaction.

When Claw needs to sign a transaction (e.g., send 80 USDC):

Step 9 — Validate spending limits

Before touching the private key, the server checks:

Is this single action under $5?
Would today's total stay under $20?
If either check fails, the transaction is rejected. The private key is never even decrypted.

Step 10 — Decrypt the private key in memory

SESSION_SECRET → scrypt → derived key → AES-GCM decrypt → raw private key. This happens entirely in RAM.

Step 11 — Sign the transaction locally

The private key reconstructs an Account object. The transaction data (recipient, amount, gas, chain ID for Base) is signed cryptographically. This produces a "signed raw transaction" — a hex blob that proves the wallet owner authorized this specific action.

Step 12 — Broadcast to the blockchain

The signed hex is sent to Base chain via an RPC call (through Alchemy). The network verifies the signature, executes the transfer, and returns a transaction hash.

Step 13 — Private key is discarded again

The function returns. The Account object and private key go out of scope and are garbage collected. The private key is back to existing only in encrypted form on disk.

If disaster strikes (server wiped, data lost):

Step 14 — Recovery

The user provides their Share B. The server has Share A. The two shares are combined using Shamir reconstruction to get the derived encryption key. That key decrypts agent_wallet.enc, recovering the private key and full wallet access.

If a guarantor was set up (2-of-3 scheme), any 2 of the 3 shares work — so even if the server's share is lost, the user + guarantor can recover together.

So the entire security model rests on one thing: SESSION_SECRET. If someone has that and the agent_wallet.enc file, they can decrypt the private key. If they have neither, the wallet is just an address on the blockchain with no way to move its funds.
